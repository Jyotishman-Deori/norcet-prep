# NORCET prep — feature-prompts implementation notes

All six sessions from `norcet-feature-prompts.md` are implemented against the
**modular (post-A1) codebase**. The spec was written for the old monolithic
`App.jsx`; every change below was re-homed to the right module and adapted to
the real data model. `npm run build` is clean after every session.

---

## Spec-vs-reality adaptations (important)

These are places where following the spec verbatim would have silently broken
things. Each was adapted:

1. **Weekly Summary field name.** `dailyHistory` entries are
   `{ date, attempted, correct }` — not `answered`. The summary sums
   `attempted`. (Verbatim spec → weekly counts always 0.)
2. **Daily reminder already existed** as `data.preferences.dailyReminder =
   { enabled, time, lastNotified }` with a working Settings card + `maybeNotify`
   effect. Per the spec's own conflict-check, it was **not** reimplemented —
   `maybeNotify` now also calls `pushNotification()` to log into the inbox.
3. **`data.history[qId]` has no `topic`.** The weekly weakest/improved-topic
   logic resolves topic from `allQuestions` (the `getWeakTopics` pattern).
4. **Theme context, not module globals.** No module-level `T`/`IS_DARK`; screens
   use `useTheme()` (`{ theme: T, isDark: IS_DARK }`).
5. **`getISOWeek`** lives once in `lib/utils.js`, imported by Home + App.
6. **Vite, not CRA.** Client env is `import.meta.env.VITE_VAPID_PUBLIC_KEY`
   (the spec's `process.env.REACT_APP_*` does not exist here).
7. **Generated service worker.** This project uses `vite-plugin-pwa`
   (`generateSW`), so there is no hand-written `public/sw.js`. Push handlers
   live in `public/push-sw.js` and are pulled in via `workbox.importScripts`.
8. **Notification → quiz routing.** Notification/weekly "Start" actions route
   through `handleHomeNavigate` so a quiz spec actually *starts* a quiz
   (`startQuiz`) instead of just switching `nav.screen`.

## Session 5 — manual setup (cannot be tested in this environment)

Push is **safe-by-default**: with no VAPID key set, all push helpers no-op and
the app behaves exactly as before. To turn it on:

1. `npm install` (pulls the new `web-push` + `@vercel/kv` deps).
2. Generate keys: `npx web-push generate-vapid-keys`.
3. Add a **Vercel KV** (Upstash Redis) store to the project — this auto-adds
   `KV_REST_API_URL` / `KV_REST_API_TOKEN`.
4. Set env vars (see `.env.example`):
   - Client (build-time): `VITE_VAPID_PUBLIC_KEY`
   - Server: `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT`
5. Deploy. The cron in `vercel.json` runs `/api/send-reminders` daily at
   14:00 UTC (~19:30 IST). Adjust the schedule as needed (note: Vercel Hobby
   allows one cron/day; finer per-user timing would need an hourly cron +
   storing each user's timezone — a future enhancement).

Files: `api/subscribe.js`, `api/active.js`, `api/send-reminders.js`,
`public/push-sw.js`, plus `vite.config.js` / `vercel.json` / `package.json`.
